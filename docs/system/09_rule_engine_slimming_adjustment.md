# 09_rule_engine_slimming_adjustment.md

## 1. 文档目的

本文档用于冻结 **当前阶段个人单用户工具** 下，规则层进一步瘦身的方向。

它不是推翻现有规则层，而是在已经完成：

- 轻画像 / 重画像分层
- 两阶段 AI
- 决策层语义收口

之后，对规则层职责再做一次收口，解决当前最明显的问题：

- 拦不住很多真正无关通知
- 却容易误杀开放机会类通知

---

## 2. 当前问题定义

基于 50 条公文通真实样本的多轮重放，当前规则层暴露出两个核心问题：

1. **对复杂无关通知的拦截能力并不强**
   - 外学院奖学金公示
   - 内部会议 / 部门推进
   - 教职工服务
   - 部门工作通知

   这类通知通常仍会先落到 `unknown`，再交给 AI 或决策层处理。

2. **对开放机会类通知存在过度误杀**
   - 创新创业短课
   - 通识课
   - 研习营 / 夏令营
   - 校级招募 / 展示活动

   这些通知往往没有明确受众排除，但会被课程名、学院名、身份词等启发式提前打成 `irrelevant`，导致 AI 根本没有机会接手。

---

## 3. 重新定义规则层在当前阶段的定位

在当前阶段，规则层不应再被视为“主要筛选器”。

更准确的定位应是：

**规则层 = 便宜的护栏层 + 显式信号提取层**

它的主要职责是：

1. 挡掉最明确的硬失配通知
2. 提取后续 AI 需要的结构化显式信号
3. 决定是否值得进入 AI
4. 决定 AI 需要看哪些画像 facet

它不再负责：

1. 复杂受众推理
2. 开放机会最终相关性判断
3. 基于复杂画像做细粒度机会匹配
4. 通过大量启发式去替代 AI 的自然语言理解

---

## 4. 新的规则层硬原则

### 4.1 只有“明确排除”才允许硬筛

规则层只能在以下情况直接输出 `irrelevant`：

- 通知明确限定身份，且当前用户不匹配
  - 例如：`仅限研究生`、`仅限教职工`
- 通知明确限定学院 / 专业 / 年级 / 届别，且当前用户不匹配
  - 例如：`仅限材料学院本科生`
- 通知明确限定课程受众，且当前用户不匹配
  - 例如：`已选修《大学英语》的同学`

### 4.2 没有明确排除时，不允许因为“未命中”而硬筛

以下情形不能直接作为 `irrelevant`：

- 发布学院不是当前用户学院
- 正文出现某学院名，但没有明确说明受众
- 正文出现课程名，但没有明确说明“已选课学生”
- 正文出现“本科生 / 研究生 / 毕业生”等词，但只是背景描述，不是明确受众约束

### 4.3 `publisher` 不是 `explicit_audience`

必须冻结：

- `publisher` 只说明“谁发的”
- `explicit_audience` 才说明“发给谁”

规则层只允许基于 `explicit_audience` 做学院 / 专业 / 身份硬筛，不允许基于 `publisher` 做硬筛。

---

## 5. 开放机会保护带

当前阶段应明确存在一类“优先保留给 AI”的通知。

若通知满足以下任一类型，且没有明确受众排除当前用户，则规则层应优先输出 `unknown`，而不是 `irrelevant`：

- 创新创业短课
- 研习营 / 夏令营
- 通识课 / 公开课程 / 已上线课程
- 讲座 / 展示活动 / 招募活动
- 校级报名 / 申请 / 选拔通知

这些通知的特点是：

- 可能跨学院开放
- 可能不写明目标受众
- 需要结合用户复杂画像才能判断有没有价值

因此它们不应该由规则层直接拍死。

---

## 6. 规则层应继续保留的能力

这轮瘦身不是让规则层失去价值，而是让它保留最有成本收益比的能力：

1. 显式身份硬失配
2. 显式学院 / 专业 / 年级硬失配
3. 显式课程硬失配
4. 动作词提取
5. 截止时间提取
6. 显式受众提取
7. `should_invoke_ai`
8. `required_profile_facets`

---

## 7. 规则层应主动收回的逻辑

建议主动收回或明显弱化以下逻辑：

1. 仅因为正文出现身份词就当成明确受众
2. 仅因为正文出现课程名就当成“当前课程通知”
3. 仅因为发布单位与当前学院不一致就做负向推理
4. 对开放机会做过度负向启发式
5. 在规则层继续扩张复杂语义推理

---

## 8. 最小实施点

本轮不建议大改模块结构，只建议在现有规则层中做最小增量收口：

### 8.1 `audience_matcher.py`

重点调整：

- 收紧 `explicit identity` 提取
- 收紧 `explicit course` 识别
- 明确区分：
  - 课程通知（停课 / 调课 / 已选课受众）
  - 开放课程机会（上线 / 开放 / 通识 / 短课）

### 8.2 `rule_configs.json`

重点新增或调整：

- 开放机会保护规则
- 不把开放机会直接推成 `relevant`
- 只负责保留候选状态并声明 `required_profile_facets`

### 8.3 测试

至少覆盖：

1. 创新创业短课 -> `unknown`
2. 通识课上线 -> `unknown`
3. 研习营 / 夏令营 -> `unknown`
4. 仅限研究生 -> `irrelevant`
5. 明确课程停课通知 + 课程不匹配 -> `irrelevant`

---

## 9. 与 AI 层的衔接

规则层瘦身后的目标不是“放弃筛选”，而是把筛选顺序改成：

1. 规则层挡掉明确无关
2. 规则层保留开放机会
3. AI Stage 1 做轻画像粗筛
4. AI Stage 2 做重画像精筛

也就是说：

**当前阶段要把“误杀开放机会”的风险从规则层移走，而不是把复杂理解重新塞回规则层。**

---

## 10. 验收标准

规则层瘦身完成后，应满足：

1. 开放机会类通知不再被规则层大面积误杀
2. 明确身份 / 学院 / 课程硬失配仍然能直接拦截
3. `publisher` 与 `explicit_audience` 的边界更清楚
4. 规则层没有继续扩张成重型 NLP 层
5. 与当前两阶段 AI 架构衔接更顺

---

## 11. 一句话结论

**当前阶段规则层最合理的形态，不是承担主要相关性判断，而是只拦最明确的无关通知，把开放机会和复杂通知保留下来交给 AI。**
